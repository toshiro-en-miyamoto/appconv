P: Philosophy

P1. Inherit the behaviors and appearnces rather than the source code.

Reason The primary aim of application conversions is to reimplement behaviors and appearances of the application rather than to rewrite the legacy logics with a different programming language.

Example, good Copy behaviors and appearances such as:

  • how GUI screens are organized
  • how GUI screens and gudgets look like
  • how GUI gudgets respond
  • how printed sheets look like
  • how APIs available for external systems respond

Example, bad Don't bring the dark past such as:

  • how components/modules of the application are organized
  • how variables/classes and subroutines/methods are organized
  • how lines of code represent logic

Example It is preferable to redesign database layouts in synch with the new programming design, but you have to inherit the legacy layouts when:

  • other systems access the database
  • data migration is going to cost you
If the legacy layouts must be brought to the new application, you might want to
  • design new layouts anyway, then
  • implement a wrapper layer to absorb the differences between the legacy layouts and new one.

P2: At the very beginning, specify behaviors and appearances of the target application.

Reason Without a comprehensive understanding of behaviors and appearances of the target application, you cannot copy them into the new application.

Example, good Start a conversion project by preparing a compreheinsive set of testcases to fully specify behaviors and appearances of the target application. Advantages of having such testcases include:

  • the Test Driven Development can be employed
  • you don't have to write detail design documents (only if stakeholders agree)
  • the testcases can be used in the test phase to prove that the new application behave as the same way as the legacy one.
Note that some design documents are required to write including but not restricted to
  • GUI screen hierarchy diagrams
  • component diagrams
  • datastore layouts

Example There might exist some specification documents of the target application. However, many long-lived applications have been modified and enhanced without updating its specification documents accordingly, so that documents (even if any) cannot be fully trusted. Thus, you will end up with conducting some level of expemental operations with the application to know behaviors and appearances of the application. Why not leave the experiments in documents as testcases?

Example Some projects may manually analyse source code files of the target application to specify its behaviors and appearances in documents. However, many long-lived applications are likely to be all patchwork due to fixing bugs by commenting out old lines and squeezing new functionalities between lines. Thus, you may find it hard to understand the target application only by reading through its source code, and may have to conduct some level of expemental operations to see how the application behaves. Why not leave the experiments in documents as testcases?
